\section{Double-Entry Ledger Architecture}

ByteVault implements an immutable, double-entry ledger to guarantee absolute financial integrity. Every financial event (transfer, fee, deposit) is recorded as a balanced \texttt{journal\_entry} comprising multiple \texttt{journal\_lines}.

\subsection{Core Ledger Concept}
The fundamental rule of a double-entry system is that money is never created or destroyed, only moved between accounts. Thus, for any transaction, the sum of all lines must exactly equal zero.

\[
\sum_{i=1}^{n} \text{amount\_cents}_i = 0
\]

\textbf{Why it is there:} Legacy banking systems often rely on a simple \texttt{balance} column that is incremented or decremented. If a race condition occurs or a transaction fails midway, finding out \textit{why} the balance is wrong is impossible. The ledger provides a perfect, replayable history of all financial events.

\subsection{Ledger Service \& Journal Posting}
All ledger interactions are funneled through the \texttt{LedgerService}. To prevent floating-point precision errors, the ledger strictly operates in \textbf{cents} (\texttt{amountCents} as integers).

\subsection{Maker/Checker Approvals \& Fund Holds}
High-value operations require a Maker (who initiates) and a Checker (who approves). When a Maker requests a transfer, the funds are not immediately moved in the ledger. Instead, an \texttt{account\_hold} is placed.

\begin{lstlisting}[style=typescript]
// Snippet from transfers.ts (Maker Route)
const available = await getAvailableCustomerBalanceCents({ client, accountId: from.accountId });
if (available.availableCents < amountCents) {
  throw new Error('Insufficient funds');
}

const holdRes = await client.query(
  `INSERT INTO account_holds (account_id, amount_cents, reason, status)
   VALUES ($1, $2, 'Transfer Request', 'ACTIVE') RETURNING id`,
  [from.accountId, amountCents]
);
\end{lstlisting}

\textbf{Why it is there:} This prevents the customer from spending the same funds while the Checker is reviewing the request. The \texttt{availableCents} calculation dynamically subtracts active holds from the actual ledger balance.

\subsection{Immutable Reversals}
Journal entries are \textbf{append-only}. Once posted, an entry cannot be UPDATED or DELETED. If a mistake is made, a \textbf{Reversal Entry} is posted.

\begin{lstlisting}[style=sql]
-- Enforcing single reversal via schema
ALTER TABLE journal_entries 
ADD CONSTRAINT unique_reversal 
UNIQUE (reversal_of_entry_id);
\end{lstlisting}

\textbf{Why it is there:} Immutability ensures audit trails are never tampered with. The unique constraint guarantees that a malicious or buggy employee cannot reverse the same erroneous transaction twice, which would result in creating artificial funds.

\subsection{Inter-Branch Clearing}
During a 2PC transfer, money moves across database boundaries. Since the ledger must balance locally, we use an \texttt{INTERNAL} ledger account named \texttt{CLEARING\_INTERBRANCH}.
\begin{itemize}
    \item \textbf{Main DB}: Debits Customer A, Credits Clearing Account.
    \item \textbf{Sub DB}: Debits Clearing Account, Credits Customer B.
\end{itemize}
When both transactions commit, the global sum across the organization balances perfectly, while local ledgers remain compliant.
